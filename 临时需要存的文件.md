### 1. 针对“双重去重与幂等性”
**面试官潜台词：** “你既然用了 SQLite，为什么还要加个内存 Set？这不是多此一举吗？”
*   **必问问题：** “请详细描述一下你的双重去重流程。如果服务重启，内存里的 Set 清空了，SQLite 是怎么兜底的？在高并发下，频繁查数据库会不会有性能瓶颈？”
*   **你需要吃透的点：**
    *   **短路求值逻辑：** 必须能画出 `if msg_id in memory_set: return` -> `if db.exists(msg_id): return` -> `process()` -> `save to db & set` 的流程图。
    *   **Cursor 的作用：** 明白 Cursor 是微信 API 的“分页书签”，它和去重是两回事（Cursor 负责不漏消息，去重负责不重复处理）。
    *   **性能权衡：** 知道 Set 是 O(1) 复杂度，而 DB 是 I/O 操作。内存是第一道防线，挡掉 99% 的重复请求。

### 2. 针对“多线程异步与 Flask”
**面试官潜台词：** “Flask 不是同步框架吗？你开那么多线程，服务器内存扛得住吗？”
*   **必问问题：** “在 Flask 中启动后台线程处理业务，如果瞬间涌入 1000 个请求，你的系统会发生什么？原生线程（`threading.Thread`）和线程池（`ThreadPoolExecutor`）有什么区别？你为什么选择前者？”
*   **你需要吃透的点：**
    *   **GIL 锁的影响：** 明白 Python 的多线程在 I/O 密集型任务（如调 API、查库）下是有效的，但在 CPU 密集型下受限。
    *   **资源耗尽风险：** 承认原生线程在极端高并发下的风险（内存爆炸），并准备好改进方案（“如果上生产，我会改用线程池并设置 `max_workers`”）。
    *   **守护线程（Daemon）：** 理解 `thread.daemon = True` 的含义（主进程退出时，子线程强制结束）。

### 3. 针对“RAG 两阶段检索”
**面试官潜台词：** “为什么要搞这么复杂？直接向量检索不行吗？重排序慢不慢？”
*   **必问问题：** “你的 `initial_k=10` 和 `top_n=5` 是怎么拍脑袋决定的？重排序（Rerank）模型通常比向量检索慢很多，这会增加用户等待时间吗？如果 Rerank 服务挂了，你的降级逻辑具体是怎么写的？”
*   **你需要吃透的点：**
    *   **召回率 vs 准确率：** 向量检索快但可能不准（召回），Rerank 准但慢（精排）。10 选 5 是为了在速度和精度间找平衡。
    *   **代码级降级：** 能够口述出 `try...except` 或者 `if rerank_service is None` 的逻辑，证明你真的写了容错代码。
    *   **安全召回（Safety Check）：** 解释为什么要把向量第一名强行塞回去（防止 Rerank 模型因为语义理解偏差把最相关的文档给删了）。

### 4. 针对“MinIO/OSS 配置化切换”
**面试官潜台词：** “你是怎么做到改个配置就能换存储的？用了设计模式吗？”
*   **必问问题：** “请展示一下你的存储层代码结构。你是如何抽象 MinIO 和 OSS 的接口的？如果我要增加一个阿里云 NAS 存储，你的代码需要改动哪些地方？”
*   **你需要吃透的点：**
    *   **适配器模式/策略模式：** 即使你没写标准的 Design Pattern，也要能说出一套统一的接口（比如都有 `upload_file`, `get_url` 方法）。
    *   **环境变量/配置文件：** 说明你是如何通过 `.env` 或 `config.py` 里的开关来决定初始化哪个客户端的。

### 5. 针对“全自动化入库流水线”
**面试官潜台词：** “解析 PDF 是很脏很累的活，你是怎么处理乱码、图片和表格的？”
*   **必问问题：** “你的‘多格式解析流水线’具体包含哪些步骤？如果用户上传了一个 100MB 的扫描版 PDF，你的系统会卡死吗？你是如何处理解析失败的文档的？”
*   **你需要吃透的点：**
    *   **解析器选型：** 知道 PyPDF2、MinerU 或其他工具各自的优缺点（比如 MinerU 对公式和表格支持更好）。
    *   **异常处理：** 能够说出你在 `main.py` 或 `oss_event_handler.py` 里是如何捕获解析异常并记录日志的。
    *   **进度管理：** 再次联系到 `Cursor Manager` 或类似的状态记录，证明你的流水线是可监控的。
